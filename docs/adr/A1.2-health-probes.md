# A1.2 — ADR: Health Probes

**Status:** Accepted  
**Date:** 2025-12-01  
**Deciders:** Architecture team  
**Context:** Wave 1 — Foundation & Security

## Context

Container orchestrators (Kubernetes, Docker Swarm) require health endpoints to
determine whether a service is alive and ready to accept traffic. The SOVD
standard does not define health endpoints, but operational deployments need them.

## Decision

Expose a `GET /sovd/v1/health` endpoint returning structured JSON with system
metrics (CPU, memory, uptime, component connectivity status).

| Aspect | Decision | Rationale |
|--------|----------|-----------|
| **Endpoint path** | `/sovd/v1/health` (under SOVD prefix) | Consistent with the rest of the API; avoids a separate management port |
| **Health model** | `native-health` crate with `HealthMonitor` | Decoupled from SOVD logic; reusable across deployment modes |
| **Metrics** | CPU %, memory %, uptime, component count | Sufficient for K8s liveness/readiness; extended in W2.1 (system-info) |
| **Response code** | 200 = healthy, 503 = degraded | Standard convention for load balancer health checks |

## Consequences

- **Positive:** Kubernetes-native deployment without custom probes
- **Positive:** `HealthMonitor` is reusable for system-info (W2.1) and bridge heartbeat (W3.1)
- **Implementation:** `native-health/src/lib.rs`, `native-sovd/src/routes.rs` (health_check handler)
