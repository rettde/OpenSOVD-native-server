# A2.3 — ADR: Secrets Abstraction

**Status:** Accepted  
**Date:** 2026-01-15  
**Deciders:** Architecture team  
**Context:** Wave 2 — Operational Hardening

## Context

The server needs secrets for JWT signing, API key validation, TLS certificates,
and (future) database credentials. Storing secrets in plaintext TOML configuration
is acceptable for development but fails security audits in production.

## Decision

Implement a `SecretsProvider` trait abstracting secret retrieval. Provide two
implementations:

| Implementation | Use case |
|----------------|----------|
| **EnvSecretsProvider** | Reads secrets from environment variables (default, 12-factor compatible) |
| **FileSecretsProvider** | Reads secrets from files (Kubernetes Secrets volume mounts) |

Configuration in TOML points to the secret source:

```toml
[auth]
jwt_secret_env = "SOVD_JWT_SECRET"    # EnvSecretsProvider
# or
jwt_secret_file = "/run/secrets/jwt"   # FileSecretsProvider
```

| Aspect | Decision | Rationale |
|--------|----------|-----------|
| **No plaintext secrets in config** | Secrets always referenced by env var name or file path | Security audit compliance; no accidental git commits of secrets |
| **Trait-based** | `SecretsProvider` trait | Extensible to HashiCorp Vault, AWS Secrets Manager, etc. |
| **Validation** | Startup validation checks that referenced secrets are non-empty | Fail-fast; no runtime surprises from missing secrets |

## Consequences

- **Positive:** Production-ready secret management out of the box
- **Positive:** Extensible to cloud secret managers without changing server code
- **Implementation:** `native-interfaces/src/secrets.rs` — trait + env/static providers
- **Tests:** 8 unit tests
