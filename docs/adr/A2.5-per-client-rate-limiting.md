# A2.5 — ADR: Per-Client Rate Limiting

**Status:** Accepted  
**Date:** 2026-01-15  
**Deciders:** Architecture team  
**Context:** Wave 2 — Operational Hardening

## Context

The SOVD server is exposed to multiple clients (diagnostic tools, fleet management
platforms, ML pipelines). Without rate limiting, a single misbehaving client can
monopolize server resources and starve other consumers. This is especially critical
for batch export endpoints (Wave 4) that can generate large responses.

## Decision

Implement per-client rate limiting as a tower middleware layer using a sliding window
algorithm with configurable parameters:

```toml
[rate_limit]
enabled = true
max_requests = 100
window_secs = 60
```

| Aspect | Decision | Rationale |
|--------|----------|-----------|
| **Granularity** | Per client IP (or per API key when auth is enabled) | Prevents single-client monopolization without penalizing the fleet |
| **Algorithm** | Sliding window counter (DashMap-based) | Simple, memory-efficient, no external dependencies |
| **Configuration** | TOML config, disabled by default | Development-friendly; production enables via config |
| **Response** | 429 Too Many Requests with `Retry-After` header | Standard HTTP rate limiting convention |
| **Scope** | All SOVD endpoints (not health/metrics) | Health probes must never be rate-limited |

## Consequences

- **Positive:** Protects server from client-side DoS (accidental or malicious)
- **Positive:** Per-client fairness; one client's burst doesn't affect others
- **Negative:** Adds ~1μs overhead per request (DashMap lookup)
- **Implementation:** `native-sovd/src/rate_limit.rs` — `RateLimiter` struct + middleware
- **Tests:** 5 unit tests
