# A3.4 — ADR: BridgeTransport Trait

**Status:** Accepted  
**Date:** 2026-02-15  
**Deciders:** Architecture team  
**Context:** Wave 3 — Cloud Bridge, Multi-Tenant, Variant-Aware

## Context

The cloud bridge mode (W3.1) enables a cloud-hosted SOVD gateway to forward
diagnostic requests to on-premise SOVD servers via an intermediary transport.
The transport mechanism varies by deployment:

- **Development/testing:** In-memory channel (same process)
- **Production:** WebSocket, gRPC, MQTT, or proprietary OEM tunnel
- **Edge:** Direct TCP with TLS mutual authentication

Hardcoding a specific transport would prevent OEM customization and limit
deployment flexibility.

## Decision

Define a `BridgeTransport` trait abstracting the session lifecycle and message
forwarding:

```rust
#[async_trait]
pub trait BridgeTransport: Send + Sync {
    async fn create_session(&self, target: &str) -> Result<BridgeSession, BridgeError>;
    async fn forward(&self, session_id: &str, request: &[u8]) -> Result<Vec<u8>, BridgeError>;
    async fn heartbeat(&self, session_id: &str) -> Result<(), BridgeError>;
    async fn disconnect(&self, session_id: &str) -> Result<(), BridgeError>;
    fn list_sessions(&self) -> Vec<BridgeSession>;
}
```

| Aspect | Decision | Rationale |
|--------|----------|-----------|
| **Trait location** | `native-interfaces/src/bridge.rs` | Alongside other provider traits (ComponentBackend, EntityBackend) |
| **Default impl** | `InMemoryBridgeTransport` in `native-sovd` | Sufficient for testing and single-binary deployments |
| **Session model** | `BridgeSession` with ID, target, status, timestamps | Enables session listing, monitoring, and cleanup |
| **Error type** | `BridgeError` enum (ConnectionFailed, SessionNotFound, Timeout, etc.) | Typed errors for proper HTTP status mapping |
| **Feature gating** | Bridge routes enabled via `config.bridge.enabled` | Zero overhead when bridge is disabled |

## Alternatives Considered

| Option | Pros | Cons |
|--------|------|------|
| **Separate bridge binary** | Clean process isolation | Deployment complexity; two binaries to manage |
| **Generic HTTP proxy** | No custom transport code | Loses session semantics; no heartbeat/disconnect lifecycle |
| **gRPC-only transport** | Efficient binary protocol | Forces gRPC dependency on all deployments |

## Consequences

- **Positive:** OEMs implement their tunnel protocol behind `BridgeTransport`
- **Positive:** Bridge mode is feature-gated; disabled deployments have zero overhead
- **Positive:** Session lifecycle (create, heartbeat, disconnect) maps cleanly to REST endpoints
- **Implementation:** `native-interfaces/src/bridge.rs` (trait), `native-sovd/src/bridge.rs` (InMemoryBridgeTransport + REST handlers)
- **Tests:** 8 integration tests
