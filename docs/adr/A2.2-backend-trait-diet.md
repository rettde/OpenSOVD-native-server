# A2.2 — ADR: ComponentBackend Trait Diet

**Status:** Accepted  
**Date:** 2026-01-15  
**Deciders:** Architecture team  
**Context:** Wave 2 — Operational Hardening

## Context

The `ComponentBackend` trait started with ~15 methods in Wave 1. Each wave adds new
capabilities (KPIs, bulk operations, software packages, modes). Without discipline,
the trait grows to 30+ methods, making every mock/test backend painful to implement
and every new backend integration a large effort.

## Decision

Keep `ComponentBackend` as the single backend trait but apply the following diet rules:

| Rule | Rationale |
|------|-----------|
| **Default implementations** | New methods get `fn method() -> Result<_, DiagServiceError> { Err(NotSupported) }` defaults so existing backends don't break |
| **Separate concern traits** | `ExtendedDiagBackend` for UDS vendor extensions (x-uds routes); `EntityBackend` for apps/funcs | 
| **No KPI/health methods** | System-level metrics live in `HealthMonitor`, not in `ComponentBackend` |
| **Bulk as extension** | `bulk_read` / `bulk_write` have defaults returning `NotSupported`; backends opt in |

## Alternatives Considered

| Option | Pros | Cons |
|--------|------|------|
| **Split into many small traits** | Each trait is focused | Combinatorial explosion; complex trait bounds on `AppState` |
| **Feature-flag methods** | Only compiled when needed | Cfg complexity; hard to test all combinations |
| **Keep everything in one trait** | Simple dispatch | 35+ methods; mock backends become 200+ lines of boilerplate |

## Consequences

- **Positive:** New backends only implement the methods they support; defaults return `NotSupported`
- **Positive:** Vendor-specific extensions isolated in `ExtendedDiagBackend`
- **Positive:** Test mocks stay concise (only override methods under test)
- **Implementation:** `native-interfaces/src/backend.rs` — trait definitions with defaults
