# ADR A3.5 — API Versioning Contract

**Status:** Accepted
**Date:** 2026-03-19
**Context:** Enterprise integrators need stability guarantees before building fleet infrastructure against the SOVD API.

---

## Decision

### URL Versioning

- **Current:** `/sovd/v1/…` — stable, production-ready
- **Breaking changes** require a new version prefix: `/sovd/v2/…`
- Both versions may coexist during a deprecation period (minimum 6 months)
- The server advertises supported versions via `GET /sovd/v1` response (`supportedVersions` array)

### Deprecation Policy

1. **Announcement:** Breaking change announced in CHANGELOG at least one release before removal
2. **Sunset header:** Deprecated endpoints return `Sunset: <date>` HTTP header (RFC 8594)
3. **Grace period:** Minimum 6 months or 2 major releases, whichever is longer
4. **Removal:** Deprecated endpoint returns `410 Gone` with migration instructions

### What Constitutes a Breaking Change

| Change Type | Breaking? | Example |
|-------------|-----------|---------|
| New endpoint | No | `GET /sovd/v1/system-info` |
| New optional field in response | No | Adding `softwareVersion` to `SovdComponent` |
| New required field in request | **Yes** | Requiring `tenant_id` in POST body |
| Remove endpoint | **Yes** | Removing `GET /sovd/v1/components/{id}/flash` |
| Change field type | **Yes** | `faultId: string` → `faultId: integer` |
| Change HTTP status code semantics | **Yes** | 200 → 204 for existing endpoint |
| New optional query parameter | No | `?variant=base` filter |
| New middleware (transparent) | No | Rate limiting, tenant extraction |

### Extension Endpoints

- Vendor-specific endpoints use `x-` prefix: `/sovd/v1/x-uds/…`, `/sovd/v1/x-bridge/…`
- Extension endpoints are **not covered** by the versioning guarantee
- Extensions may change between minor versions with notice in CHANGELOG

### Version Negotiation

```json
// GET /sovd/v1
{
    "serverName": "opensovd-native-server",
    "serverVersion": "0.9.0",
    "sovdVersion": "1.1.0",
    "supportedVersions": ["v1"],
    "deprecatedVersions": []
}
```

## Consequences

- All current `/sovd/v1/` endpoints are now considered stable
- New Wave 3 features (bridge, tenant, variant) are added as new endpoints or optional parameters, not breaking changes
- Bridge endpoints use `/sovd/v1/x-bridge/…` prefix (extension, not stable)
- Server-info response gains `supportedVersions` and `deprecatedVersions` arrays
