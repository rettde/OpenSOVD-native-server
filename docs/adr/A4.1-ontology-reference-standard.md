# A4.1 — ADR: Ontology Reference Standard for Semantic Data Annotations

**Status:** Accepted  
**Date:** 2026-03-20  
**Deciders:** Architecture team  
**Context:** Wave 4 — Data Catalog & Batch Export

## Context

Wave 4 adds semantic metadata (`semanticRef`, `unit`, `normalRange`, `samplingHint`)
to `SovdDataCatalogEntry` and fault ontology enrichment to `SovdFault`. Downstream
ML pipelines need a stable, machine-readable reference schema to map diagnostic
signals to a shared vocabulary.

Three options were evaluated:

| Option | Description | Pros | Cons |
|--------|-------------|------|------|
| **(a) COVESA VSS** | Vehicle Signal Specification — tree-structured, YAML-based, widely adopted in AUTOSAR/SDV | Open standard, large signal catalog, tooling ecosystem, COVESA governance | Limited to signals (no service/operation semantics), not all OEM DIDs map cleanly |
| **(b) IFEX** | Interface Exchange framework — superset of VSS with service + data modeling | Richer model, covers operations + data, COVESA-adjacent | Less mature tooling, smaller community, heavier schema |
| **(c) Vendor-defined** | OEM-specific ontology with optional VSS mapping layer | Maximum flexibility, OEM-specific signals | No interoperability, every integration is custom |

## Decision

**Option (a): COVESA VSS as primary reference, with extensibility for vendor paths.**

- `semanticRef` fields use VSS dot-notation paths (e.g. `Vehicle.Powertrain.Battery.StateOfCharge`)
- Vendor-specific signals use `x-vendor.` prefix (e.g. `x-vendor.mbds.CoolantPressure`)
- IFEX can be added as a future `semanticRefType` if needed (forward-compatible)
- The `DataCatalogProvider` trait (A4.2) abstracts the ontology source, so OEMs can
  plug in their own mapping without changing the API contract

## Rationale

- VSS is the de facto standard for vehicle signal taxonomy in the SDV ecosystem
- COVESA governance ensures long-term stability and versioning
- The `x-vendor.` prefix pattern allows OEM-specific extensions without polluting the
  standard namespace
- ML pipelines can normalize on VSS paths across OEMs, enabling transfer learning
- ISO 17978-3 does not mandate a specific ontology but recommends "industry-standard
  signal references where available"

## Consequences

- All `semanticRef` values MUST be valid VSS paths or `x-vendor.*` prefixed strings
- Schema version (E4.1) tracks ontology version independently of API version
- `DataCatalogProvider` implementations are responsible for VSS path resolution
- Default implementation uses static mappings; OEM profiles can override via
  `OemProfile::data_catalog_provider()`
