# A2.1 — ADR: StorageBackend Trait

**Status:** Accepted  
**Date:** 2026-01-15  
**Deciders:** Architecture team  
**Context:** Wave 2 — Operational Hardening

## Context

Wave 2 introduces historical fault storage (W2.2) and Wave 3 requires per-tenant
namespaced storage (W3.2). Hardcoding a specific storage engine (e.g. sled) would
prevent swapping backends for different deployment scenarios (embedded vs. cloud).

## Decision

Define a `StorageBackend` trait abstracting key-value persistence. Provide two
implementations:

| Implementation | Use case |
|----------------|----------|
| `DashMap` (in-memory) | Default for development, testing, and stateless deployments |
| `sled` (on-disk) | Persistent fault storage behind `persist` feature flag |

Key design choices:

| Aspect | Decision | Rationale |
|--------|----------|-----------|
| **Trait surface** | `get`, `put`, `delete`, `scan_prefix` | Minimal KV operations sufficient for fault storage and future use |
| **Feature gating** | `persist` Cargo feature enables sled backend | Keeps binary size small for embedded; sled pulled in only when needed |
| **Namespace support** | Key prefix convention (`{tenant}/{component}/...`) | Prepares for W3.2 multi-tenant without changing the trait |
| **Error type** | `StorageError` enum | Unified error handling across backends |

## Consequences

- **Positive:** Fault persistence is optional and backend-swappable
- **Positive:** Multi-tenant namespace isolation (W3.2) works via key prefix convention
- **Negative:** Abstraction overhead vs. direct sled usage (negligible in practice)
- **Implementation:** `native-core/src/storage.rs` — trait + DashMap impl; `native-core/src/storage_sled.rs` — sled impl (feature-gated)
- **Tests:** 11 unit tests covering both backends
