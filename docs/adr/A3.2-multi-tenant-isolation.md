# ADR A3.2 — Multi-Tenant Data Isolation Strategy

**Status:** Accepted
**Date:** 2026-03-19
**Context:** Wave 3 adds multi-tenant fleet/workshop support (W3.2). Tenant isolation must be decided before building tenant-scoped state.

---

## Decision Drivers

1. Fleet diagnostics serves multiple workshops/OEMs on shared infrastructure
2. Audit logs must be tenant-isolated for compliance (ISO 27001, UNECE R155)
3. Fault state, lock ownership, and execution history are security-sensitive
4. Performance: tenant isolation should not require per-tenant process/container

## Options Considered

| # | Option | Isolation | Complexity | Performance |
|---|--------|-----------|------------|-------------|
| A | **Policy-only isolation** — single state, `AuthzPolicy` gates access | Weak (logic bugs leak data) | Low | Best |
| B | **Per-tenant state partitioning** — separate `AppState` per tenant | Strong | High (N × memory) | Worst |
| C | **Tenant-scoped storage namespacing** — single state, all keys prefixed with tenant ID | Strong (data-level) | Medium | Good |

## Decision

**Option A + C — Policy-based AuthZ (A) with tenant-scoped storage namespacing (C).**

Rationale:
- Policy-only (A) is too weak alone — a single bug in AuthzPolicy leaks cross-tenant data.
- Per-tenant state (B) is wasteful — most state (component catalog, group definitions) is shared across tenants.
- Namespacing (C) adds data-level isolation at the storage layer without duplicating in-memory structures.
- The combination of A + C gives defense-in-depth: AuthzPolicy prevents unauthorized access at the API layer, and tenant-scoped keys prevent data leakage at the storage layer.

## Architecture

### Tenant Extraction

```
HTTP Request
    │
    ├─ JWT claim: "tenant_id" (preferred)
    ├─ Header: "X-Tenant-ID" (fallback / dev mode)
    ├─ API key → tenant mapping (from config)
    │
    ▼
TenantContext { tenant_id, tenant_name, isolation_level }
    │
    ├─ Injected into request extensions (like AuthenticatedClient)
    ├─ Available to all handlers via extractor
    └─ Passed to storage/audit operations as namespace prefix
```

### Tenant-Scoped Storage Keys

| Resource | Key Pattern | Example |
|----------|-------------|---------|
| Faults | `{tenant_id}:{component_id}:{fault_id}` | `workshop-a:hpc-main:DTC-P0100` |
| Audit log | `{tenant_id}:{seq}` | `workshop-a:00042` |
| Locks | `{tenant_id}:{component_id}` | `workshop-a:hpc-main` |
| Executions | `{tenant_id}:{execution_id}` | `workshop-a:exec-uuid` |

### Isolation Levels

```rust
pub enum TenantIsolation {
    /// No isolation — single-tenant deployment (backward compatible)
    None,
    /// Namespace isolation — shared state with tenant-prefixed keys
    Namespace,
    /// Strict isolation — separate state instances per tenant (future)
    Strict,
}
```

### OemProfile Per-Tenant Selection (A3.3)

The `TenantContext` middleware can select different `OemProfile` implementations per tenant:

```rust
// Config-driven tenant → profile mapping
[tenants.workshop_a]
profile = "mbds"
isolation = "namespace"

[tenants.workshop_b]
profile = "default"
isolation = "namespace"
```

## Consequences

- New struct: `TenantContext` in `native-interfaces`
- New middleware: `tenant_context_middleware` in `native-sovd` (extracts tenant from JWT/header)
- New extractor: `TenantId` for route handlers
- FaultManager, AuditLog, LockManager get tenant-aware methods
- Default behavior (no tenant header/claim) = `TenantContext::default()` (single-tenant, backward compatible)
- JWT `Claims` struct gains optional `tenant_id` field
