# A4.2 — ADR: DataCatalogProvider Trait

**Status:** Accepted  
**Date:** 2026-03-20  
**Deciders:** Architecture team  
**Context:** Wave 4 — AI-Ready Diagnostic Data

## Context

Wave 4 enriches `SovdDataCatalogEntry` with semantic metadata (VSS path, normal range,
sampling hints, classification tags). The source of this metadata varies by deployment:

- **Open-source default:** Static metadata hardcoded or loaded from configuration
- **OEM deployments:** Metadata sourced from CDF/ODX files, proprietary ontology databases,
  or external metadata services
- **Cloud deployments:** Centrally managed metadata synced from fleet-level ontology repositories

Without an abstraction, metadata sources would be hardcoded, making OEM customization
require forking the server code.

## Decision

Introduce a `DataCatalogProvider` trait in `native-interfaces`:

```rust
pub trait DataCatalogProvider: Send + Sync {
    fn metadata(&self, component_id: &str, data_id: &str) -> Option<DataSemantics>;
    fn all_metadata(&self, component_id: &str) -> Vec<(String, DataSemantics)>;
    fn schema_version(&self) -> String;
}
```

Key design choices:

| Aspect | Decision | Rationale |
|--------|----------|-----------|
| **Trait location** | `native-interfaces` crate | Same crate as `ComponentBackend` — keeps all provider traits together |
| **Default impl** | `StaticDataCatalogProvider` with builder pattern | Zero-config for open-source users; `.add(component, data_id, semantics)` for static registration |
| **Schema versioning** | `schema_version() -> String` on the trait | E4.1 requires ML pipelines to detect schema changes; version is provider-level, not per-entry |
| **Return type** | `Option<DataSemantics>` (not `Result`) | Missing metadata is normal (not all items have semantic annotations); errors should be logged internally by the provider |
| **Thread safety** | `Send + Sync` bound | Provider is stored in `Arc<dyn DataCatalogProvider>` in `AppState` |

## Alternatives Considered

| Option | Pros | Cons |
|--------|------|------|
| **Extend `ComponentBackend` trait** | No new trait; metadata co-located with data access | Bloats an already large trait (30+ methods); not all backends know semantic metadata |
| **Configuration-only (TOML/JSON)** | Simple, no code changes for new metadata | Cannot source metadata from external services; no runtime updates |
| **Middleware enrichment** | Decoupled from backend; transparent to handlers | Complex; requires intercepting and modifying response bodies; hard to test |

## Consequences

- **Positive:** OEMs can implement `DataCatalogProvider` in their proprietary profile
  without modifying core server code. The `StaticDataCatalogProvider` covers the
  open-source use case with zero configuration overhead.
- **Positive:** Schema version tracking (E4.1) is built into the trait contract,
  not bolted on after the fact.
- **Negative:** One more trait for OEM integrators to implement (mitigated by the
  default `StaticDataCatalogProvider`).
- **Integration:** Provider is injected into `AppState.data_catalog` and used by
  snapshot export (W4.2), schema introspection (W4.4), and NDJSON export handlers.

## Implementation

- `native-interfaces/src/data_catalog.rs` — Trait definition, `DataSemantics`, `NormalRange`, `StaticDataCatalogProvider`
- `native-sovd/src/state.rs` — `AppState.data_catalog: Arc<dyn DataCatalogProvider>`
- 6 unit tests in `data_catalog.rs`
