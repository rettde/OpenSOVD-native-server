# A1.4 — ADR: Startup Configuration Validation

**Status:** Accepted  
**Date:** 2025-12-01  
**Deciders:** Architecture team  
**Context:** Wave 1 — Foundation & Security

## Context

The server reads configuration from a TOML file (`config.toml`). Invalid or
contradictory settings (e.g. TLS cert path without key path, auth enabled without
a secret) can cause subtle runtime failures that are hard to diagnose in production.

## Decision

Validate all configuration at startup before binding the HTTP listener. Fail fast
with descriptive error messages listing all validation errors at once.

| Aspect | Decision | Rationale |
|--------|----------|-----------|
| **Timing** | Before server bind | Fail-fast prevents partial startup; all errors reported together |
| **Validation scope** | TLS pair consistency, auth secret presence, port range, rate limit params | Covers all config fields that can cause runtime failures |
| **Error reporting** | Collect all errors, print all, then exit | Better UX than failing on first error (user fixes everything in one pass) |
| **Config format** | TOML with serde `AppConfig` struct | Type-safe deserialization with serde defaults for optional fields |

## Consequences

- **Positive:** No silent misconfigurations; clear error messages at startup
- **Positive:** Serde defaults make most fields optional (zero-config for development)
- **Implementation:** `native-server/src/main.rs` — `validate_config()` function
