# A4.3 — ADR: Batch Export Serialization Format

**Status:** Accepted  
**Date:** 2026-03-20  
**Deciders:** Architecture team  
**Context:** Wave 4 — Data Catalog & Batch Export

## Context

Wave 4 introduces batch diagnostic export endpoints (`/components/{id}/snapshot`,
`/export/faults`) for ML pipeline consumption. Single-resource REST polling is too
slow for training data collection. The export format is a long-lived contract with
downstream data infrastructure.

| Option | Description | Pros | Cons |
|--------|-------------|------|------|
| **(a) JSONL streaming** | Newline-delimited JSON, one record per line | Simple, streamable, no new deps, debuggable with `jq`, standard in ML pipelines (Spark, pandas) | Larger than binary, no built-in schema |
| **(b) Apache Arrow/Parquet** | Columnar binary format via IPC | Fastest for analytics, zero-copy, schema embedded | Heavy deps (`arrow` crate ~2MB), complex server-side, not human-readable |
| **(c) CSV with schema header** | Plain text tabular | Universal import, tiny | No nested types, no schema evolution, lossy for JSON values |

## Decision

**Option (a): JSONL (Newline-Delimited JSON) as primary export format.**

- Response content type: `application/x-ndjson`
- Each line is a self-contained JSON object (parseable independently)
- A metadata preamble line (first line) contains export metadata (E4.3 reproducibility)
- Streamed via chunked transfer encoding for large exports
- Future: Arrow/Parquet can be added as `Accept: application/vnd.apache.arrow.stream`
  content negotiation without breaking JSONL consumers

## Rationale

- Zero additional dependencies — `serde_json` already in the workspace
- JSONL is the de facto standard for ML data ingestion (Hugging Face datasets,
  Spark structured streaming, pandas `read_json(lines=True)`)
- Streamable: server can flush records as they're produced, reducing memory pressure
- Human-debuggable: `curl ... | jq .` works immediately
- Schema evolution is natural: new fields are additive, consumers ignore unknown fields
- The metadata preamble line provides reproducibility (E4.3) without a separate endpoint

## Format Specification

```
{"_meta":true,"exportedAt":"...","serverVersion":"0.10.0","schemaVersion":"1.0.0",...}
{"id":"battery_voltage","componentId":"hpc","value":12.6,"unit":"V","semanticRef":"Vehicle.Powertrain.Battery.Voltage"}
{"id":"coolant_temp","componentId":"hpc","value":85.2,"unit":"°C","semanticRef":"Vehicle.Powertrain.CoolantTemperature"}
```

## Consequences

- Export endpoints return `Content-Type: application/x-ndjson`
- First line is always the metadata preamble (consumers MUST skip `_meta: true` lines)
- Schema version (E4.1) is included in the preamble for contract stability
- Large exports use chunked transfer encoding (axum `StreamBody`)
- Future binary formats (Arrow) can coexist via content negotiation
