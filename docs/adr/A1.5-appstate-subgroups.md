# A1.5 — ADR: AppState Sub-Grouping

**Status:** Accepted  
**Date:** 2025-12-01  
**Deciders:** Architecture team  
**Context:** Wave 1 — Foundation & Security

## Context

The axum `AppState` struct grows with every wave: backend references, fault manager,
lock manager, audit log, health monitor, execution store, proximity store, package
store, OEM profile, rate limiter. A flat struct with 15+ fields becomes unreadable
and makes it unclear which state belongs to which domain.

## Decision

Organize `AppState` into logical sub-groups, each with its own struct:

```rust
pub struct AppState {
    pub backend: Arc<dyn ComponentBackend>,
    pub extended_backend: Arc<dyn ExtendedDiagBackend>,
    pub entity_backend: Arc<dyn EntityBackend>,
    pub diag: DiagState,        // fault_manager, lock_manager, diag_log
    pub security: SecurityState, // oem_profile, audit_log, rate_limiter
    pub runtime: RuntimeState,   // health, execution_store, proximity_store, package_store
    pub data_catalog: Arc<dyn DataCatalogProvider>,  // Wave 4
}
```

| Aspect | Decision | Rationale |
|--------|----------|-----------|
| **Grouping** | By domain (diag, security, runtime) | Natural boundaries; each sub-state has cohesive concerns |
| **Clonability** | All sub-states derive `Clone` (Arc-wrapped internals) | Required by axum's `State` extractor |
| **Extensibility** | New fields go into the appropriate sub-group | Prevents the flat-struct explosion; clear ownership |

## Consequences

- **Positive:** Readable state management; easy to find where new state belongs
- **Positive:** Sub-groups can be passed independently to helper functions
- **Negative:** Slightly more verbose access (`state.diag.fault_manager` vs `state.fault_manager`)
- **Implementation:** `native-sovd/src/state.rs`
