# ADR A3.1 — Cloud Bridge Topology

**Status:** Accepted
**Date:** 2026-03-19
**Context:** Wave 3 requires brokered remote diagnostics (W3.1). Before writing code, the bridge architecture must be decided.

---

## Decision Drivers

1. Fleet diagnostics requires cloud-to-vehicle tunneling (workshop → cloud → vehicle)
2. The SOVD Gateway already runs on-vehicle; the bridge must coexist without duplicating state
3. Deployment flexibility: same binary for edge and cloud, or separate binaries?
4. Latency: WebSocket provides persistent duplex; gRPC is more structured; MQTT is fire-and-forget

## Options Considered

| # | Option | Pros | Cons |
|---|--------|------|------|
| A | **Feature-gated bridge mode in same binary** | Single artifact, shared AppState, simpler CI | Binary size grows; edge deployments carry unused cloud code |
| B | **Separate `native-bridge` binary** | Clean separation of concerns, independent scaling | Duplicated config parsing, IPC needed for shared state |
| C | **Sidecar/proxy pattern** (e.g. Envoy + WASM filter) | No Rust code for transport; leverages existing infra | Loses type safety; complex debugging; vendor lock-in risk |

## Decision

**Option A — Feature-gated bridge mode in the same binary.**

Rationale:
- The bridge is a thin WebSocket relay layer that reuses the existing `ComponentRouter` and `AppState`. No new state management needed.
- Feature flag `bridge` keeps the code out of edge builds. Binary size delta is minimal (~50 KiB for tokio-tungstenite).
- Fleet operators deploy the same container image with `SOVD_BRIDGE__ENABLED=true` for cloud instances.
- If scaling needs diverge later, extracting into a separate binary is straightforward (the `BridgeTransport` trait ensures loose coupling).

## Architecture

### Protocol

- **Transport:** WebSocket (RFC 6455) over TLS
- **Framing:** JSON-RPC 2.0 messages wrapping SOVD requests/responses
- **Authentication:** mTLS for vehicle-to-cloud; JWT for client-to-cloud
- **Heartbeat:** WebSocket ping/pong every 30s; reconnect with exponential backoff

### `BridgeTransport` Trait (A3.4)

```rust
#[async_trait]
pub trait BridgeTransport: Send + Sync {
    async fn accept_remote(&self) -> Result<BridgeSession, BridgeError>;
    async fn forward_to_vehicle(&self, session: &BridgeSession, req: SovdBridgeRequest) -> Result<SovdBridgeResponse, BridgeError>;
    async fn heartbeat(&self, session: &BridgeSession) -> Result<(), BridgeError>;
    async fn disconnect(&self, session: &BridgeSession) -> Result<(), BridgeError>;
}
```

## Consequences

- New dependency: `tokio-tungstenite` (WebSocket), feature-gated behind `bridge`
- New module: `native-sovd/src/bridge.rs` (relay + client)
- New config section: `[bridge]` with `enabled`, `listen_addr`, `connect_url`, `heartbeat_interval_secs`
- Bridge routes are mounted only when `bridge` feature is enabled
