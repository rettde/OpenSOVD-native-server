# A3.3 — ADR: TenantContext Middleware

**Status:** Accepted  
**Date:** 2026-02-15  
**Deciders:** Architecture team  
**Context:** Wave 3 — Cloud Bridge, Multi-Tenant, Variant-Aware

## Context

Multi-tenant fleet deployments require that every request carries a tenant identity.
Tenant context must be available to all handlers for data isolation (namespace-scoped
storage keys), audit attribution, and access control. Without a standardized
extraction mechanism, each handler would need to parse tenant information independently.

## Decision

Implement `TenantContext` as a struct extracted from JWT claims (`tenant_id` field)
or from the `X-Tenant-Id` header (for API key auth). The context is injected via
axum's extractor mechanism and flows through all auth paths.

```rust
pub struct TenantContext {
    pub tenant_id: String,
    pub isolation: TenantIsolation,
}
```

| Aspect | Decision | Rationale |
|--------|----------|-----------|
| **Extraction source** | JWT `tenant_id` claim (primary), `X-Tenant-Id` header (fallback for API key auth) | JWT is the source of truth; header fallback for non-JWT auth flows |
| **Isolation model** | `TenantIsolation` enum: `Shared`, `Namespaced`, `Dedicated` | Supports progressive isolation levels per OEM policy |
| **Scope key generation** | `TenantContext::scoped_key(base)` → `"{tenant_id}/{base}"` | Transparent namespace prefixing for storage backends |
| **Missing tenant** | Falls back to `"default"` tenant | Backwards-compatible with single-tenant deployments |
| **Configuration** | `MultiTenantConfig` in TOML | Enable/disable, default isolation level, header name |

## Consequences

- **Positive:** Tenant context available everywhere without per-handler boilerplate
- **Positive:** Storage namespace isolation is automatic via `scoped_key()`
- **Positive:** Single-tenant deployments work unchanged (default tenant)
- **Implementation:** `native-interfaces/src/tenant.rs` — TenantContext, TenantIsolation, MultiTenantConfig
- **Tests:** 15 unit tests
