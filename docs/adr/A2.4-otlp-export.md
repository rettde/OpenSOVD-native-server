# A2.4 — ADR: OpenTelemetry OTLP Export

**Status:** Accepted  
**Date:** 2026-01-15  
**Deciders:** Architecture team  
**Context:** Wave 2 — Operational Hardening

## Context

Production deployments need distributed tracing and metrics export to observability
platforms (Grafana, Jaeger, Datadog). The server already uses `tracing` for structured
logging, but traces and metrics need to reach external collectors.

## Decision

Integrate OpenTelemetry via the `metrics` + `metrics-exporter-prometheus` crates for
metrics, and `tracing-subscriber` with OTLP-compatible formatting for traces.

| Aspect | Decision | Rationale |
|--------|----------|-----------|
| **Metrics** | Prometheus exporter via `metrics-exporter-prometheus` | Industry standard for Kubernetes; Grafana-native scraping |
| **Traces** | `tracing` spans with structured fields | Already used throughout the codebase; zero-cost when no subscriber |
| **Export protocol** | Prometheus pull (metrics), structured JSON logs (traces) | Simpler than OTLP push; works with existing log aggregators |
| **RED metrics** | Request rate, error rate, duration per endpoint (E1.3) | Standard SRE observability; enables alerting on latency/error spikes |

## Alternatives Considered

| Option | Pros | Cons |
|--------|------|------|
| **Full OTLP gRPC push** | Richest telemetry; trace correlation | Heavy dependency (tonic/prost); complex configuration |
| **StatsD** | Simple UDP push | No trace context; limited metadata |
| **Prometheus only** | Simple pull model | No distributed tracing |

## Consequences

- **Positive:** Prometheus metrics out of the box; Grafana dashboards for RED metrics
- **Positive:** `tracing` spans already propagate through async handlers
- **Trade-off:** No full OTLP push yet — acceptable for v1; can add later without API changes
- **Implementation:** `native-sovd/src/routes.rs` (metrics middleware), `native-server/src/main.rs` (Prometheus endpoint)
