## 1. Decision

**Extension, not Fork.**

`native-*` uses `opensovd-core` as a vanilla dependency and extends it with
its own routes, middleware, CDF, and provider logic. No upstream PR required.

The SOVD CDF Validator (`dsagmbh/sovd-cdf-validator`) accepts custom rules,
so `native-server` can validate its full SOVD compliance by providing its own
CDF (already generated in `native-sovd/src/openapi.rs`) with custom rule sets.

---

## 2. Context

The `OpenSOVD-native` project (`native-*` crates) implements a full SOVD server
with 469+ tests, covering faults, operations, modes, locks, data, auth, OEM
profiles, multi-tenancy, audit, health, UCM, TARA, RXSWIN, and UDS extensions.

The Eclipse `opensovd-core` project provides a clean, minimal framework with:
- **Provider-per-entity** pattern (traits attached to `Component`/`App` via builder)
- **Topology** (async RwLock, broadcast events, area/component/app relationships)
- **ServerBuilder** + axum routes (data endpoints, entity discovery, auth middleware)
- **Models** crate (serde + optional jsonschema)

### Why no upstream PR?

1. **CDF Validator custom rules** — native-server generates its own complete CDF
   (`openapi.rs`, 1400+ lines) with all ISO 17978-3 mandatory paths. The CDF
   validator accepts custom rules, so SOVD compliance is validated independently
   of what opensovd-core knows about.

2. **Architectural mismatch** — opensovd-core uses provider-per-entity
   (`Component.fault_provider()`), native-* uses `ComponentBackend` god-trait +
   `ComponentRouter`. Decomposing the god-trait into provider traits would require
   a fundamental rewrite of native-* for no functional benefit.

3. **Feature velocity** — native-* has 18800+ LOC and 469+ tests across
   5 domain-specific crates. Upstreaming this incrementally would create
   months of review overhead without clear benefit to the Eclipse project,
   which targets a different use case (generic SOVD framework vs. production
   diagnostic server).

4. **Independence** — native-* can evolve its features (UCM, TARA, RXSWIN,
   UDS security, OEM profiles) without waiting for upstream review cycles.

### What opensovd-core provides to native-*

| opensovd-core Feature | Used by native-* | How |
|-----------------------|-------------------|-----|
| `Topology` | Optional | Can populate opensovd Topology from ComponentRouter backends |
| `DataProvider` trait | Possible adapter | Could wrap `ComponentBackend::read_data` |
| `ServerBuilder` | Optional | native-server has its own axum wiring |
| `Authenticator`/`Authorizer` | Reference | native-* has its own auth stack |
| Discovery routes | Reference | native-* has richer entity discovery |

## 4. Architectural Differences (Reference)

### 4.1 Dispatch Pattern

```
native-*:     AppState → Arc<dyn ComponentBackend> → ComponentRouter → backend_for(id)
opensovd-*:   AppState → Topology → entity.provider() → &dyn XxxProvider
```

| Aspect | native-* | opensovd-core |
|--------|----------|---------------|
| **Dispatch** | `ComponentBackend` god-trait (~30 methods) + `ComponentRouter` | Per-concern provider traits per entity |
| **Entity model** | `SovdComponent` DTO; backend holds logic | `Component` struct owns `Box<dyn Provider>` |
| **State** | `AppState { DiagState, SecurityState, RuntimeState }` | `AppState { vendor_info, topology }` |
| **Route handlers** | Single `routes.rs` (8200+ lines) | Per-concern route modules |
| **Error model** | `SovdErrorEnvelope` (OData §9.4) | `GenericError` + `ErrorCode` |

### 4.2 Why the patterns coexist

The god-trait pattern (`ComponentBackend`) is well-suited for diagnostic gateways
that forward requests to external ECUs/CDAs — a single backend object manages all
capabilities for a set of components.

The provider-per-entity pattern is suited for static topology with individually
configured entities. Both are valid SOVD implementations.

native-* does not need to adopt opensovd-core's provider pattern. The CDF
validator validates the API surface, not the internal dispatch mechanism.

## 5. CDF Validation Strategy

### 5.1 Generated CDF

`native-sovd/src/openapi.rs` generates a complete OpenAPI 3.1 CDF with:
- All ISO 17978-3 mandatory paths (components, data, faults, operations, modes, locks, configurations, logs)
- Correct HTTP methods and status codes per SOVD section
- `x-sovd-version`, `x-sovd-applicability` extensions
- OEM-customizable via `CdfPolicy` trait
- 12 contract tests validating CDF structure

### 5.2 CDF Validator Custom Rules

The `dsagmbh/sovd-cdf-validator` supports custom rule sets:
- **Standard rules**: validate ISO 17978-3 mandatory paths and schemas
- **Custom rules**: validate vendor-specific extensions (RXSWIN, TARA, UCM, x-uds)

native-server provides both:
1. Standard CDF paths pass the default SOVD validation rules
2. Extension paths (`/rxswin`, `/tara/*`, `/ucm/*`, `/x-uds/*`) are validated
   with custom rules specific to native-server

### 5.3 No upstream change needed

Since the CDF is self-describing (OpenAPI spec) and the validator is rule-driven,
native-server's SOVD compliance is verified independently. opensovd-core does
not need to know about faults, operations, or modes for native-server to pass
CDF validation.

## 6. Key Decisions

### D1: opensovd-core stays vanilla
No fork, no PR. native-* depends on opensovd-core as a published crate
(or git dependency). If opensovd-core adds features (e.g. FaultProvider),
native-* can optionally adopt them.

### D2: native-* owns all domain features
Faults, operations, modes, locks, auth, OEM profiles, audit, health,
UCM, TARA, RXSWIN — all implemented and maintained in `native-*` crates.

### D3: CDF compliance via custom rules
The CDF validator with custom rules proves ISO 17978-3 conformance.
No need to align internal architecture with opensovd-core's patterns.

### D4: Optional future contributions
If opensovd-core's community expresses interest in specific features
(e.g. fault models, OEM profiles), native-* can contribute them as
standalone PRs. This is a community-driven decision, not a technical
necessity.

### D5: Reference branch preserved
The `inc/native-server` branch on `rettde/opensovd-core` is kept as a
reference for the provider-per-entity decomposition (FaultProvider PoC).
It demonstrates how native-* features could be contributed if desired.

## 8. Relationship to opensovd-core

| Aspect | Relationship |
|--------|-------------|
| **Dependency** | native-* MAY depend on opensovd-core crates (optional) |
| **CDF** | native-* generates its own CDF, validated independently |
| **Contribution** | Optional, community-driven, not architecturally required |
| **API surface** | Both implement ISO 17978-3 mandatory endpoints |
| **Internal dispatch** | Different patterns (god-trait vs provider-per-entity) |
| **Coexistence** | Both can run as SOVD servers; CDF validator proves compliance for each |
