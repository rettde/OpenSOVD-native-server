# A1.1 — ADR: Graceful Shutdown

**Status:** Accepted  
**Date:** 2025-12-01  
**Deciders:** Architecture team  
**Context:** Wave 1 — Foundation & Security

## Context

The SOVD server must handle shutdown signals (SIGTERM, SIGINT) without dropping
in-flight diagnostic requests. UDS/DoIP operations can take several seconds
(e.g. ECU flash, long-running routines). Abruptly terminating connections risks
leaving ECUs in inconsistent states.

## Decision

Use `tokio::signal` to catch OS signals and axum's `graceful_shutdown` with a
configurable drain timeout (`server.shutdown_timeout_secs` in TOML config).

| Aspect | Decision | Rationale |
|--------|----------|-----------|
| **Signal handling** | `tokio::signal::ctrl_c()` + Unix SIGTERM | Standard for containerized deployments (Kubernetes sends SIGTERM) |
| **Drain period** | Configurable via TOML, default 30s | Allows in-flight UDS operations to complete; K8s `terminationGracePeriodSeconds` alignment |
| **New connections** | Rejected during drain | Prevents new work from extending shutdown |

## Consequences

- **Positive:** Safe for ECU flash operations; no orphaned UDS sessions
- **Positive:** Kubernetes-ready out of the box
- **Implementation:** `native-server/src/main.rs` — signal handler + axum graceful shutdown
